# Windows 4.1.9 研究路线（公开执行已禁用）

当前 `0.1.0-rc1` 不提供 Windows 程序版本切换。`legacy-plan` 只返回：

```text
DISABLED_PENDING_LEGACY_HARDENING
```

统一入口不再暴露下载、准备、切换、旧版捕获、恢复或清理命令；直接执行 `legacy_windows.py` 的敏感子命令也会拒绝。Codex 不得修改脚本、设置旁路开关或建议用户手工降级。

## 为什么禁用

历史上，维护者曾用腾讯签名的 Windows 微信 4.1.9.57 完成临时切换、只读捕获，并恢复到当时的当前版本。该事实只能说明研究路线曾可行，不能证明公开工具在断电、进程终止、磁盘错误或本地 payload 被修改时可安全恢复。

2026-07-15 发布审查确认至少缺少：

- 修改程序文件前持久化、可重入的事务日志和独占锁；
- 每个变更边界的故障注入测试；
- “临时文件 → 签名/哈希 → 原子替换”的统一恢复；
- 恢复命令对中间/失败状态的完整支持；
- 对将要复制的完整 payload 树逐文件校验，而非只验启动器和一个 DLL；
- 对提取工具本身的可信来源验证；
- 能覆盖旧客户端可能修改的全部用户数据，而不是把有限数据库快照描述成完整回滚。

这些问题没有被“多次确认”或“历史真机成功”消除。为保证陌生用户把链接交给 Codex 时不会自动改变微信程序文件，本路线从首个公开稳定版候选中退出。

## 重新启用的最低门槛

只有在以下条件全部具备后，才考虑通过新的审查重新开放：

1. 完整状态机、锁、事务日志、原子替换和幂等恢复实现；
2. 完整 payload manifest 与复制前后复核；
3. 覆盖断电/kill/写入失败的自动故障注入；
4. 独立 Windows 测试机完成切换、捕获、恢复、重启登录和数据完整性检查；
5. 明确的用户数据全量备份/恢复边界；
6. 每个敏感阶段继续要求当前对话单独批准；
7. 新版本经过安全复审，不与默认当前版只读捕获自动串联。

当前 Windows 支持面只有 [当前版本只读恢复](windows.md)。如果它返回兼容性失败，应停止、保留脱敏错误码并等待适配，不应降级验证标准。
